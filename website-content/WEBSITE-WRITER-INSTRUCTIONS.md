# FlowRunner Integrations — Website Content Brief

**Audience of this document:** the Claude Code project that writes the FlowRunner website, blog, and marketing content.

**What you're getting:** a machine-readable catalog of every FlowRunner integration (`integrations-index.json` + `integrations/<slug>.json`), plus the positioning, taxonomy, page templates, and SEO playbook to turn it into an integrations marketing engine.

**The one-sentence strategy:** we have **329 verified integrations exposing 7,382 actions**, each already built and documented as an **AI-agent tool**. Don't market this as "we have a lot of nodes like n8n." Market it as **the largest verified tool library for building AI agents and automations** — and mint a landing page for every single integration (the n8n/Zapier SEO playbook, done AI-native).

> **Always pull live numbers from `integrations-index.json`** (`totalIntegrations`, `totalActions`, `totalTriggers`, `oauthServices`, `selfHostedServices`). The catalog is regenerated as integrations are added — never hardcode counts in templates.

## 2. Positioning — the three pillars

Lead every surface with these, in this order of emphasis.

### Pillar 1 — "The AI-agent tool library" (the wedge)
This is the differentiator, and it's a 2026 narrative, not a me-too one.

- Every integration was **built as an AI Agent Tool**: rich, structured descriptions on each of the 7,382 actions, typed parameters, and sample results — exactly the metadata an LLM needs to call a tool correctly.
- The story: **"Give your AI agent 329 production-ready tools — or wire them into deterministic automations. Same catalog, both worlds."**
- Competitors (n8n, Zapier, Make) are **workflow-first**; FlowRunner can own **agent-tools-first**. Workflow automation is table stakes; being the tool layer for agents is the growth story.
- Concrete claims you can make from the data:
  - "7,382 callable actions across 329 services."
  - "Every action carries an AI-readable description and sample output."
  - Sub-narrative for developers: these map almost 1:1 to **tool schemas / MCP tools** — position FlowRunner as the place to get a huge, curated tool surface without hand-writing integrations.

### Pillar 2 — "Verified against the live API" (the trust moat)
A quality claim competitors can't cheaply copy.

- Every integration was **spec'd from the vendor's own API reference**, and the catalog went through a **reference-accuracy audit** where each service was re-checked against the live API — catching and fixing services that were on wrong or deprecated endpoints.
- Say it responsibly: **"Built and verified against each vendor's official API"** / **"Accuracy-audited against live vendor documentation."**
- **Do not** claim "tested against every live production account" — the audit was against official API references and machine-verified signing/schemas, not a live call to all 329. Keep the claim to *specified-and-verified-against-the-official-API*, which is true and strong.

### Pillar 3 — "Breadth with depth" (the proof)
Numbers + complete categories.

- **329 integrations / 7,382 actions / 71 services with real-time or polling triggers / 80 self-hostable.**
- Depth, not just count: for the AI providers, FlowRunner covers the provider's **full current API surface** (not the thin "chat" node most tools ship) — call this out on the AI category hub.
- Complete verticals (see personas below): the whole security stack, the whole identity stack, every major database, etc.

**Voice for all three:** confident, technical, concrete. AI-native (talk about agents, tools, and workflows interchangeably). Avoid breathless adjectives; let the numbers and the specificity carry it.

## 4. The per-integration landing page (the core SEO asset)

**Mint one page per integration: `/integrations/<slug>`.** That's 329 long-tail pages — the single highest-ROI move, because ~80% of the content already exists in each record. This is the n8n/Zapier playbook, modernized.

### URL & metadata
- **URL:** `/integrations/<slug>` (use the record's `slug` verbatim).
- **`<title>`:** `{name} Integration — Connect {name} to AI Agents & Workflows | FlowRunner`
- **Meta description:** derive from `shortDescription`, ≤155 chars, lead with an action verb and the name.
- **H1:** `{name} Integration`
- **schema.org:** emit `SoftwareApplication` (the FlowRunner connector) and/or `HowTo` for the recipe section; include the logo, category, and an `applicationCategory`.

### Page skeleton (map fields → sections)
1. **Hero** — logo (`iconFile`), `name`, one-line value prop derived from `shortDescription`, a "Connect {name}" CTA, and quick-fact chips: **`{actionCount} actions`**, **`{triggerCount} triggers`** (only if `hasTriggers`), **`auth`**, **`Self-hosted ✓`** (if `selfHostedCapable`).
2. **What you can do** — render `idealUseCases` as the benefit bullets (they're already outcome-phrased).
3. **AI-agent angle (the differentiator — put it high)** — "Use {name} as a tool for your AI agent." Pull 2–3 `actions` with their descriptions to show how an agent would call them. This section is what makes the page AI-native rather than a Zapier clone.
4. **Actions** — a searchable/grouped table of `actions` (group by the action's `category`). Show name + description. This is *unique, keyword-rich content per page* — great for SEO, and it's real.
5. **Triggers** — if `triggerCount > 0`, list `triggers` ("Start a workflow when …"). Emphasize that FlowRunner has **real-time + polling triggers**, not just actions (a differentiator vs. many tool libraries).
6. **Recipes / templates** — render `agentIdeas` as starter workflows. **Cross-link each named companion service to its own landing page** (e.g. an idea mentioning "Slack" links to `/integrations/slack`). This builds the internal link graph automatically.
7. **Auth & setup** — one line from `auth` (OAuth2 vs API key/token) and, if `selfHostedCapable`, "Works with self-hosted/on-prem instances."
8. **Related integrations** — 4–6 others from the same `marketingCategory`.
9. **CTA footer** — start free / book demo.

### Rules
- **Ground everything in the record.** Never list an action/trigger that isn't in `actions`/`triggers`. Never claim an auth method not in `auth`.
- **Rewrite, don't paste.** The `shortDescription`/`idealUseCases` are engineering-toned; rewrite into benefit-led marketing copy, but keep the facts.
- **Each page must be substantively unique** — the per-integration `actions` list guarantees that; don't collapse pages to a shared boilerplate.

## 7. Template / recipe gallery (activation + more SEO)

Every record's `agentIdeas` is a ready-made cross-service workflow. Aggregate them into a **`/templates`** (or `/recipes`) gallery — the #1 activation surface for automation products (users copy a template instead of starting from blank), and hundreds more indexable pages.

- Each recipe page: the workflow, the 2–3 integrations it uses (linked to their landing pages), and a "use this template" CTA.
- Cluster by persona/category. Dedupe near-identical ideas.
- These double as blog fodder ("5 ways to automate X with AI").

## 9. Voice, claims, and guardrails

**Do:**
- Talk in terms of **agents, tools, actions, triggers, and workflows** interchangeably — that's the AI-native register.
- Be specific and numeric; cite real action names and counts.
- Say **"verified/built against each vendor's official API."**
- Note **self-hosting** and **real-time triggers** where the data supports them (`selfHostedCapable`, `hasTriggers`) — both are differentiators.

**Don't:**
- Don't claim live end-to-end testing of every integration (see Pillar 2 wording).
- Don't invent capabilities, actions, or triggers beyond the record.
- Don't lead with "329 nodes like n8n" — lead with the tool-library/agent framing.
- Don't overclaim MCP if the product doesn't expose it yet — position it as "agent-ready tools" and let the MCP angle be aspirational/roadmap unless engineering confirms it ships.

## 10. Keeping content in sync

The catalog is generated deterministically from the extensions repo:

```bash
# in the flowrunner-extensions repo
python3 website-content/generate-catalog.py
```

- Re-run after any batch of new/changed services; copy the refreshed `integrations-index.json` + `integrations/` into the website project.
- New categories only appear if a new vendor doesn't match the existing keyword map — if `categorySource` is `"uncategorized"` (bucket `"Other"`) for any record, add its vendor to `KEYWORD_CATEGORY` in the generator (or let the writer assign it manually). As of this build, **zero** records are uncategorized.
- Treat the JSON as the **source of truth for facts** (actions, auth, counts) and your prose as the marketing layer on top.
